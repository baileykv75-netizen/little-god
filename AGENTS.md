# AGENTS.md

本文件是 Little God 项目的长期开发规则。任何自动化代理或协作者开始修改代码前，都必须先阅读本文件以及 `docs/MVP_SPEC.md`、`docs/CURRENT_TASK.md`、`docs/HANDOFF.md`。

## 1. 产品原则

1. 游戏核心是“设定规则并观察涌现结果”，不是传统升级、战斗或数值碾压。
2. 玩家是全能但非全知的创造者；乐趣来自实验、取舍、发现和世界历史。
3. 每项新功能都必须强化至少一项：可观察性、因果清晰度、实验空间、情感故事。
4. 任何系统若必须依赖大量内容才能成立，先缩小为可独立验证的版本。

## 2. 范围纪律

1. 只执行 `docs/CURRENT_TASK.md` 中列出的任务。
2. 未经明确要求，不得提前加入文明、宗教、科技树、战斗装备、联机、账号、服务器或付费系统。
3. 一个阶段完成后停止，更新 `docs/HANDOFF.md`，等待人工试玩结论。
4. 不以“顺便重构”为理由扩大改动范围。

## 3. 技术约束

1. 当前阶段使用原生 HTML、CSS、JavaScript，不引入框架和构建工具。
2. 仓库根目录必须可作为静态站点直接部署。
3. 模拟采用固定步长或受控步长；速度切换不能改变生态规则本身。
4. 模拟状态、规则配置、渲染和界面交互应保持清晰边界。
5. 所有随机行为必须能被未来的种子随机数替换；禁止把随机调用散落到难以追踪的位置。
6. 数量上限和边界检查必须存在，避免实体无限增长或浏览器卡死。
7. 不使用外部图片、字体或音频依赖，除非任务明确要求。

## 4. 交互与可读性

1. 玩家必须能看懂当前选择的物种、世界是否运行、时间速度和任务进度。
2. 物种颜色、形状和行为至少有两种区分方式，不能只依赖颜色。
3. 所有重要结果都要同时体现在画面、统计或世界纪事中。
4. 玩家操作必须有即时反馈；无效操作应给出原因。
5. 桌面端优先，但布局不得在窄屏下完全失效。

## 5. 代码质量

1. 变量和函数使用清晰英文命名；用户可见文本使用中文。
2. 避免超大函数；更新、行为决策、碰撞、绘制、UI同步应分段组织。
3. 常量集中定义，禁止在行为逻辑中散布难以解释的魔法数字。
4. 事件监听器只能注册一次；重置世界不得重复绑定。
5. 修改后至少手动验证：启动、暂停、速度切换、投放三类物种、重置、物种灭绝与重新投放。

## 6. 完成标准

一个阶段只有在以下条件全部满足时才算完成：

- 页面无控制台致命错误；
- 当前任务中的验收项全部可操作验证；
- 文档与代码状态一致；
- `docs/HANDOFF.md` 已记录完成内容、已知问题和下一步建议；
- 没有偷偷加入下一阶段功能。
